# Story 0.4 — L2 Lakehouse: Pipeline dbt Bronze/Silver/Gold e Customer 360

**Epic:** 0 — Fase 0: Ambiente e Demo
**Status:** Ready

## Descricao

Implementar o pipeline de dados dbt com tres camadas (Bronze/Silver/Gold) sobre ClickHouse como warehouse principal e DuckDB para analise local/exploracao. Construir o modelo central Customer 360 na camada Gold, cobrindo as quatro dimensoes: identidade, jornada, financeiro e comportamento. Expor data freshness visivel para o usuario final.

Este e o coracao da proposta de valor do Elexia Cortex: transformar dados fragmentados de multiplas fontes em uma visao unificada do cliente. A demo precisa mostrar um Customer 360 real populado com dados sinteticos antes de conectar fontes reais.

## Criterios de Aceitacao

- [ ] AC1: ClickHouse instalado no EKS (namespace `cortex-data`) com particionamento por `tenant_id`; schema `bronze`, `silver`, `gold` criados
- [ ] AC2: DuckDB configurado como engine de exploracao local; arquivos Parquet sincronizados do ClickHouse para DuckDB via job periodico
- [ ] AC3: Camada Bronze: tabelas raw por fonte (ex: `bronze.hubspot_contacts`, `bronze.shopify_orders`); dados chegam exatamente como enviados pelo conector Airbyte, sem transformacao
- [ ] AC4: Camada Silver: dados limpos, tipados, com PII tokenizado; joins basicos aplicados; deduplicacao simples por chave natural da fonte
- [ ] AC5: Camada Gold: modelo `gold.customer_360` com colunas: `customer_id` (UUID), `tenant_id`, dimensao identidade (nome, email_token, cpf_token, telefone_token, canais), dimensao jornada (primeiro contato, ultima interacao, touchpoints), dimensao financeiro (ltv, ticket_medio, total_pedidos, inadimplencia), dimensao comportamento (frequencia, recencia, produtos_preferidos, score_engajamento)
- [ ] AC6: dbt models com testes de qualidade: `not_null` e `unique` para chaves primarias; `accepted_values` para campos enum; `relationships` para foreign keys
- [ ] AC7: Data freshness: metadado `last_updated_at` em cada tabela Gold; endpoint `/api/data-freshness` retorna timestamp da ultima atualizacao por fonte e por entidade
- [ ] AC8: Pipeline executa em menos de 5 minutos para dataset de demo (ate 10.000 registros por tenant)

## Riscos

- **ClickHouse no EKS com Spot Instances:** interrupção de Spot pode perder dados em trânsito não persistidos; mitigação: usar `ReplicatedMergeTree` com ao menos 2 réplicas e persistent volumes.
- **dbt adapter para ClickHouse:** o adapter `dbt-clickhouse` é menos maduro que o `dbt-bigquery`; mitigação: fixar versão estável e testar todos os materializações antes de avançar.
- **PII tokenization na camada Silver:** falha no serviço de tokenização (story 0.2) bloqueia todo o pipeline; mitigação: circuit breaker com fallback para mascaramento por `***` em ambiente de teste.
- **Customer 360 ID determinístico:** colisões de SHA256 são teoricamente possíveis com datasets muito grandes; mitigação: adicionar `tenant_id` ao hash para reduzir probabilidade; monitorar colisões.

## Notas Tecnicas

- ClickHouse: usar `altinity/clickhouse-operator` para Kubernetes; engine `ReplicatedMergeTree` com `ORDER BY (tenant_id, customer_id, created_at)`
- dbt: versao 1.7+; adapter `dbt-clickhouse`; executar via `dbt run --profiles-dir .` no CI/CD
- Particao por tenant: toda tabela tem coluna `tenant_id` STRING como primeira coluna no `ORDER BY`; usar `WHERE tenant_id = ?` em todas as queries para garantir isolamento
- Customer 360 ID: `SHA256(tenant_id || ':' || email_normalizado)` como UUID deterministico antes de ter identity resolution
- Data freshness: adicionar macro dbt `{{ dbt_utils.current_timestamp() }}` em `updated_at` de cada model Gold
- Referencia: https://docs.getdbt.com/docs/build/materializations
