# Story 0.5 — L2 Identity Resolution: Matching Multi-Fonte e PPRL

**Epic:** 0 — Fase 0: Ambiente e Demo
**Status:** Draft
**Prioridade:** P1
**Estimativa:** L
**Dependencias:** 0.4 (pipeline Bronze/Silver/Gold e modelo Customer 360)

---

## Descricao

Implementar o motor de identity resolution do Elexia Cortex: identificar que registros de diferentes fontes de dados (HubSpot, Shopify, Omie, etc.) representam o mesmo cliente real, mesmo quando os dados nao sao identicos. O matching deve suportar correspondencia exata por email e CPF/CNPJ, alem de correspondencia fuzzy por nome + telefone. Implementar PPRL (Privacy-Preserving Record Linkage) para garantir que o matching possa ser feito sem expor dados em claro.

Identity resolution e o diferencial tecnico mais dificil de replicar: sem ela, o Customer 360 e apenas uma colecao de registros fragmentados. Com ela, o Cortex sabe que "wyndson@empresa.com" no HubSpot e "Wyndson Oliveira" no Shopify e o mesmo cliente.

## Criterios de Aceitacao

- [ ] AC1: Matching deterministico por email normalizado (lowercase, trim, remover aliases `+tag`) com confianca 1.0
- [ ] AC2: Matching deterministico por CPF/CNPJ (normalizado: apenas digitos) com confianca 1.0
- [ ] AC3: Matching probabilistico por nome + telefone: algoritmo de similaridade (Jaro-Winkler ou Levenshtein) com threshold configuravel (padrao: 0.85); confianca proporcional ao score
- [ ] AC4: PPRL implementado: antes do matching, PII e convertida em Bloom Filter ou Hash com chave secreta no Vault; matching opera sobre hashes, nunca sobre dados em claro
- [ ] AC5: Tabela `silver.identity_clusters` criada: `cluster_id` (UUID), `tenant_id`, lista de `source_record_ids` agrupados, `confidence_score`, `match_reason` (EMAIL | CPF | FUZZY_NAME_PHONE)
- [ ] AC6: Deduplicacao: quando dois registros sao matched, o Customer 360 consolida atributos por prioridade configuravel (ex: email do HubSpot > email do Shopify)
- [ ] AC7: Metrica de cobertura: endpoint `/api/identity/stats` retorna percentual de registros matched vs. singletones por tenant
- [ ] AC8: Testes com dataset sintetico: pelo menos 3 cenarios cobertos (match exato, match fuzzy, falso positivo detectado)

## Escopo

**IN (incluido nesta story):**
- Matching deterministico (email, CPF/CNPJ)
- Matching probabilistico (nome + telefone, Jaro-Winkler)
- PPRL com Bloom Filter ou HMAC hash
- Tabela `silver.identity_clusters`
- Consolidacao de atributos por prioridade
- Endpoint de estatisticas de matching
- Testes com dados sinteticos

**OUT (fora desta story):**
- Matching por endereco fisico (Fase 1)
- Matching por device fingerprint (Fase 1)
- Graph de identidade completo com Neo4j (Fase 1)
- Machine learning para matching (Fase 1 — requer dados historicos de feedback)
- Matching cross-tenant (nao permitido — cada tenant e isolado)

## Notas Tecnicas

- PPRL: implementar Bloom Filter sobre CPF e email com chave secreta por tenant armazenada no Vault; falso positive rate < 1% com filter size adequado
- Jaro-Winkler: usar biblioteca Python `jellyfish`; normalizar nome antes: lowercase, remover acentos, remover titulos honorificos (Dr., Sr., etc.)
- Prioridade de fontes: configuravel por tenant em `configs/identity-priority.yaml`; padrao: HubSpot > Salesforce > Shopify > Omie > QuickBooks
- Pipeline: executar como dbt model na camada Silver com `incremental` materialization para processar apenas novos registros
- Threshold fuzzy: comecar com 0.85; expor como parametro configuravel por tenant para ajuste pos-demo
- Referencia PPRL: https://arxiv.org/abs/1703.00617
