## Descricao

Implementar o AI Engine de Nivel 1 do Elexia Cortex: tres modelos preditivos operacionais — Churn Predictor, Customer Health Score e Anomaly Detection. Os modelos devem funcionar com dados sinteticos para a demo e com dados reais assim que os conectores Wave 1A estiverem ativos. MLflow gerencia o ciclo de vida dos modelos (versioning, tracking de experimentos, serving). O Churn Predictor usa XGBoost com score de 0 a 100 e top 3 motivos explicados em linguagem natural.

Esta story e o que diferencia o Elexia Cortex de um BI tradicional: em vez de apenas mostrar o que aconteceu, o Cortex prediz o que vai acontecer. Ver um score de churn de 87 com "motivos: reducao de 60% no uso nos ultimos 30 dias, atraso em 2 pagamentos, diminuicao de engajamento com suporte" na demo e o momento em que o prospect diz "eu preciso disso".

## Criterios de Aceitacao

- [ ] AC1: MLflow instalado no EKS (namespace `cortex-ai`); UI acessivel via ingress interno; experimentos organizados por modelo e versao
- [ ] AC2: Churn Predictor implementado com XGBoost: features extraidas do Customer 360 (recencia, frequencia, engajamento, pagamentos), score 0-100 (0 = sem risco, 100 = certeza de churn); threshold configuravel (padrao 70 = alto risco)
- [ ] AC3: Churn Predictor retorna top 3 motivos por cliente: lista de features com maior contribuicao SHAP, convertidas para frases em linguagem natural (ex: "Reducao de 60% no uso nos ultimos 30 dias")
- [ ] AC4: Customer Health Score calculado por regra ponderada (nao ML puro): score 0-100 composto de pagamentos em dia (30%), engajamento recente (25%), uso do produto (25%), satisfacao NPS se disponivel (20%); componentes visiveis no dashboard
- [ ] AC5: Anomaly Detection usando Isolation Forest: detecta anomalias em series temporais de comportamento do cliente (pico/queda abrupta em pedidos, pagamentos fora do padrao); alertas gerados quando anomalia e detectada com confidence > 0.8
- [ ] AC6: Todos os modelos operacionais com dados sinteticos: ao carregar dados sinteticos da story 0.14, modelos executam e produzem predicoes para todos os clientes do dataset de demo
- [ ] AC7: Predicoes salvas na tabela `gold.ai_predictions`: `customer_id`, `tenant_id`, `prediction_type` (CHURN | HEALTH | ANOMALY), `score`, `confidence`, `top_reasons_json`, `predicted_at`
- [ ] AC8: Endpoint de predicao: `GET /api/ai/predictions/{customer_id}` retorna predicoes mais recentes; `POST /api/ai/predict/batch` processa batch de clientes
- [ ] AC9: Retreino automatico: job agendado para retreinar modelos semanalmente quando novos dados chegam; MLflow registra nova versao e compara performance com versao anterior

## Riscos

- **AUC baixo com dados sintéticos:** dados gerados artificialmente podem não ter os padrões de churn reais necessários para treinar um modelo com AUC >= 0.75; mitigação: validar qualidade das features sintéticas antes do treinamento e ajustar parâmetros do gerador (story 0.14).
- **SHAP com XGBoost em produção:** cálculo de SHAP values pode ser lento para batches grandes; mitigação: pré-calcular SHAP values no momento do retreino e armazenar em `gold.ai_predictions`, não calcular on-demand.
- **MLflow no EKS sem storage persistente:** se o PVC do MLflow for perdido, o histórico de experimentos é perdido; mitigação: configurar MLflow com S3 como artifact store e PostgreSQL como metadata store.
- **Dependência circular 0.11 ↔ 0.14:** 0.11 pode começar com dados mock antes de 0.14, mas a validação final depende dos dados sintéticos de qualidade; mitigação: definir dataset mínimo de mock para desenvolvimento inicial e usar 0.14 para validação final.

## Notas Tecnicas

- XGBoost: versao 2.0+; features para churn: `days_since_last_order` (R), `order_count_90d` (F), `total_revenue_90d` (M), `payment_delays_count`, `support_tickets_count`, `product_usage_score`
- SHAP: `shap.TreeExplainer(model).shap_values(X)` para features mais impactantes; mapear top 3 features para templates de frases em portugues
- Isolation Forest: `sklearn.ensemble.IsolationForest`; treinar por serie temporal de 30 dias; `contamination=0.05` (5% de anomalias esperadas)
- MLflow tracking: logar `mlflow.log_metrics({'auc': auc, 'f1': f1})`; logar modelo com `mlflow.xgboost.log_model`
- Serving: usar MLflow Model Serving ou implementar endpoint FastAPI com modelo carregado em memoria
- Performance minima aceitavel: Churn AUC >= 0.75 com dados sinteticos (com dados reais, espera-se 0.80+)
- Templates de frases para SHAP: dicionario `feature_name → template_pt_br` (ex: `days_since_last_order → "Sem compras ha {value} dias"`)
