## Descricao

Configurar o conector Shopify no Airbyte para ingerir dados de e-commerce: Products, Orders, Customers e Refunds. Complementar com webhooks Shopify para capturar eventos em tempo quase-real (novos pedidos, pagamentos, cancelamentos). Mapear todos os campos relevantes para o modelo Customer 360, enriquecendo as dimensoes de jornada e financeiro.

Shopify e a fonte mais importante para verticais de Varejo/E-commerce, que representam uma parcela significativa do mercado-alvo do Cortex. A combinacao de dados transacionais (pedidos, pagamentos) com dados comportamentais (produtos visualizados, carrinhos abandonados) torna o Customer 360 especialmente rico para essa vertical.

## Criterios de Aceitacao

- [ ] AC1: Conector Shopify configurado no Airbyte com autenticacao via API Key + Password (Custom App); streams habilitados: `customers`, `orders`, `products`, `refunds`, `abandoned_checkouts`
- [ ] AC2: Sincronizacao completa executada com sucesso; dados em `bronze.shopify_customers`, `bronze.shopify_orders`, `bronze.shopify_products`, `bronze.shopify_refunds`
- [ ] AC3: Sincronizacao incremental configurada usando cursor `updated_at` para orders e customers
- [ ] AC4: Webhooks Shopify configurados para os topicos: `orders/create`, `orders/updated`, `orders/paid`, `orders/cancelled`, `customers/create`, `customers/update`; endpoint de recepcao no Cortex processando eventos em menos de 30s
- [ ] AC5: Mapeamento para Customer 360: `shopify_customers.email` → identidade; pedidos → dimensao financeiro (LTV, ticket medio, total pedidos); produtos frequentes → dimensao comportamento
- [ ] AC6: Documentacao de field mapping em `docs/integrations/shopify-field-mapping.md`
- [ ] AC7: Teste de ponta a ponta: criar pedido no Shopify sandbox, confirmar que aparece no `gold.customer_360` em menos de 5 minutos (via webhook) ou em menos de 4h (via sync scheduled)

## Riscos

- **Endpoint de webhook exposto publicamente:** o endpoint `POST /api/webhooks/shopify` precisa estar acessível pela internet; mitigação: validar HMAC em todas as requisições antes de qualquer processamento e usar allowlist de IPs Shopify.
- **Shopify Development Store com dados limitados:** sandbox pode não refletir todos os campos de uma loja real; mitigação: complementar com dados sintéticos da story 0.14 para cobrir os campos ausentes.
- **Streams `abandoned_checkouts` com volume alto:** pode gerar grande volume de eventos em lojas reais; mitigação: configurar filtro de data nos streams para limitar ao período relevante em ambiente de demo.
- **Dependência de story 0.6:** qualquer instabilidade no Airbyte bloqueia este conector; mitigação: garantir saúde do Airbyte antes de iniciar 0.7.

## Notas Tecnicas

- Shopify Custom App: criar em `Partners > Apps > Create app`; permissoes necessarias: `read_customers`, `read_orders`, `read_products`, `read_inventory` (read-only)
- Webhook endpoint: `POST /api/webhooks/shopify`; validar HMAC SHA-256 no header `X-Shopify-Hmac-Sha256` antes de processar
- Mapeamento financeiro: `LTV = SUM(orders.total_price) WHERE financial_status = 'paid'`; `ticket_medio = AVG(orders.total_price)`; inadimplencia = `orders.financial_status = 'unpaid'`
- Campos de PII do Shopify: `email`, `phone`, `first_name + last_name` → tokenizar na camada Silver antes de persistir no Customer 360
- Shopify sandbox: usar Shopify Development Store para testes sem afetar loja real
- Referencia: https://docs.airbyte.com/integrations/sources/shopify
