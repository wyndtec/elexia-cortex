## Descricao

Implementar a camada de seguranca base do Elexia Cortex: criptografia em repouso com AWS KMS (TDE), gerenciamento de segredos com HashiCorp Vault, tokenizacao de dados PII (CPF, email, telefone) para conformidade com LGPD, RBAC com cinco niveis de acesso, e audit log imutavel com hash-chaining para rastreabilidade completa de acoes.

Esta story e critica para viabilizar o processo de venda enterprise: clientes do porte de bancos e retailers grandes exigem evidencia de seguranca antes de assinar. O ambiente de demo precisa demonstrar seguranca real, nao apenas aparente.

## Criterios de Aceitacao

- [x] AC1: AWS KMS CMK (Customer Managed Key) criada em sa-east-1; todos os volumes EBS e buckets S3 do projeto usam esta CMK para criptografia em repouso
- [x] AC2: HashiCorp Vault instalado no EKS (namespace `cortex-infra`) em modo HA (3 replicas); auto-unseal configurado via AWS KMS; Vault UI acessivel internamente
- [x] AC3: Tokenizacao de PII implementada como servico interno: CPF, email e telefone sao tokenizados antes de qualquer persistencia; token e deterministico (mesmo input = mesmo token) e reverso apenas por servico autorizado com acesso ao Vault
- [x] AC4: RBAC implementado com cinco roles: Owner, Admin, Manager, Analyst, Viewer; permissoes documentadas por role; integrado com autenticacao (JWT ou OIDC)
- [x] AC5: Audit log imutavel hash-chained: cada evento registra hash do evento anterior (chain), timestamp, user, acao, recurso afetado; nenhuma entrada pode ser deletada ou modificada sem quebrar a chain
- [x] AC6: Teste de integridade da chain: script que valida que nenhum registro do audit log foi adulterado
- [x] AC7: Secrets da aplicacao (API keys, DB passwords) gerenciados exclusivamente pelo Vault — zero hardcoded secrets no codigo ou ConfigMaps

## Riscos

- **Vault HA no EKS:** falha no auto-unseal pode deixar o Vault selado após reinicialização de pod; mitigação: testar ciclo de restart do Vault em ambiente de staging antes do demo.
- **Tokenização determinística + LGPD:** se a chave HMAC for comprometida, todos os tokens são reversíveis; mitigação: rotação de chave documentada no runbook + acesso restrito ao Vault.
- **Complexidade do RBAC:** cinco roles com middleware Hono.js pode introduzir bugs de permissão silenciosos; mitigação: testes de integração cobrindo todos os níveis de acesso.
- **Performance do audit log hash-chaining:** inserções frequentes com hash encadeado podem criar gargalo em alta carga; mitigação: inserções assíncronas com fila em memória para ambiente de demo.

## Notas Tecnicas

- Vault: usar chart oficial `hashicorp/vault` no Helm; storage backend recomendado: Integrated Storage (Raft) para simplicidade
- Tokenizacao: implementar como microservico Python (FastAPI) com endpoints `/tokenize` e `/detokenize`; usar HMAC-SHA256 com chave no Vault para garantir determinismo
- RBAC: definir permissoes em YAML, aplicar via middleware no Hono.js (L4 API)
- Audit log: tabela `audit_events` com colunas: `id`, `previous_hash`, `current_hash`, `timestamp`, `user_id`, `action`, `resource`, `payload_json`
- Hash: SHA-256 sobre concatenacao de `previous_hash + timestamp + user_id + action + resource`
- Referencia LGPD para tokenizacao: Art. 12 — dados anonimizados e pseudoanonimizados
