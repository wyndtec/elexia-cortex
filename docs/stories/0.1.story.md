# Story 0.1 — Infraestrutura Base: AWS, EKS, Terraform e CI/CD

**Epic:** 0 — Fase 0: Ambiente e Demo
**Status:** Ready
**Prioridade:** P1
**Estimativa:** XL
**Dependencias:** nenhuma

## Descricao

Provisionar toda a infraestrutura base do Elexia Cortex em AWS sa-east-1 usando Terraform como IaC. O objetivo e ter um cluster Kubernetes (EKS) operacional com suporte a multi-tenancy por namespace, alem de um pipeline de CI/CD basico que permita deploys continuos e seguros dos servicos da plataforma.

Esta story e o alicerce de toda a Fase 0. Nenhuma outra story pode ser executada sem que o ambiente AWS esteja provisionado e o EKS ativo. Todos os servicos — Airbyte, DuckDB, ClickHouse, MLflow, Cube.dev — sao hospedados neste cluster.

## Criterios de Aceitacao

- [ ] AC1: Modulos Terraform criados para VPC, subnets publicas/privadas, security groups e IAM roles minimas em sa-east-1
- [ ] AC2: Cluster EKS (v1.29+) provisionado com node groups configurados (min 2, max 10 nodes, tipo m5.xlarge ou equivalente)
- [ ] AC3: Namespaces Kubernetes criados para cada tenant (padrao: `cortex-demo`, `cortex-prod`) com Resource Quotas e LimitRanges definidos
- [ ] AC4: Ingress Controller (NGINX ou AWS ALB Ingress) instalado e funcional, com TLS terminando no ingress
- [ ] AC5: Repositorio Git conectado ao pipeline CI/CD (GitHub Actions ou ArgoCD); deploy automatizado em ambiente de demo ao fazer push na branch `main`
- [ ] AC6: `terraform plan` e `terraform apply` executam sem erros em ambiente limpo
- [ ] AC7: Cluster acessivel via `kubectl` com kubeconfig gerado automaticamente pelo Terraform output
- [ ] AC8: Logs de infraestrutura enviados para CloudWatch com retencao de 30 dias

## Escopo

**IN (incluido nesta story):**
- Modulos Terraform: VPC, EKS, IAM, Security Groups, ECR, Route53 (hosted zone)
- Node groups EKS com auto-scaling
- Namespaces Kubernetes com isolamento por tenant
- Ingress Controller com TLS
- Pipeline CI/CD basico (lint, build, push ECR, deploy EKS)
- CloudWatch logging

**OUT (fora desta story):**
- Configuracao de Vault (story 0.2)
- Banco de dados RDS ou ElastiCache (nao necessario na Fase 0)
- Dominio customizado de producao
- WAF e Shield (escopo de producao, nao demo)
- Monitoring avancado com Prometheus/Grafana (pode ser adicionado em Fase 1)

## Riscos

- **Custo AWS inesperado:** EKS + EC2 m5.xlarge podem gerar custo elevado se o auto-scaling subir sem supervisão; mitigação: alertas de billing no CloudWatch e uso de Spot Instances para ambiente de demo.
- **Compatibilidade Terraform ↔ EKS:** versões de módulos podem divergir com atualizações; mitigação: fixar versões exatas no `terraform.lock.hcl`.
- **OIDC GitHub Actions:** configuração incorreta pode bloquear deploys; mitigação: testar pipeline em branch de feature antes de configurar `main`.
- **Multi-tenancy Kubernetes:** Resource Quotas mal calibradas podem impedir pods de subirem; mitigação: definir quotas conservadoras e ajustar pós-deploy.

## Notas Tecnicas

- Terraform state no S3 com lock via DynamoDB
- EKS Managed Node Groups com Spot Instances para reducao de custo no ambiente de demo
- Usar `eks-managed-node-groups` do modulo oficial `terraform-aws-modules/eks`
- Namespace strategy: `cortex-{tenant-id}` para isolamento real em producao; `cortex-demo` para ambiente de demonstracao
- CI/CD: preferir GitHub Actions com OIDC para autenticacao AWS (sem long-lived credentials)
- Referencia: https://registry.terraform.io/modules/terraform-aws-modules/eks/aws/latest
