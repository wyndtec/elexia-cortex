Hierarquia e Taxonomia de Dados

Como classificar, nomear e explicar cada camada da arquitetura de dados — do bruto ao inteligente. Scripts por público, analogias e respostas a objeções para a demonstração.

M
O Mapa Completo — Da Fonte ao Insight
6 camadas em ordem de refinamento — cada uma com um papel único
Princípio fundamental
Dados não têm valor no bruto. O valor está na capacidade de acessar, entender e agir. Cada camada adiciona um nível de refinamento — de bits crus a decisões de negócio. O Cortex implementa todas as 6 camadas numa plataforma unificada.
6
Camada de Apresentação — Dashboard / BI
apps/web · design/preview/
O que o usuário final vê e interage. KPIs, gráficos, alertas, chat com IA. Transforma números em decisão.
"O painel que o seu time abre de manhã."
5
Camada de Inteligência — AI Engine / LLM
services/ai-engine · intelligence.html
Modelos de ML e LLMs rodando sobre dados já curados. Geração de insights, anomalias, previsões, linguagem natural.
"A IA que responde suas perguntas de negócio."
4
Camada Semântica — Metric Catalog
metrics.html · dbt metrics · cube.dev
Onde colunas técnicas viram métricas de negócio. orders.sum(amount) → "Receita Líquida". Define a fonte única de verdade para cada número.
"O dicionário oficial dos números da empresa."
3
Data Lakehouse — Storage + Compute
lakehouse.html · Apache Iceberg · Delta Lake · DuckDB
Dados transformados, limpos e indexados. Combina o custo de storage de um lake com a performance de query de um warehouse. O coração computacional da plataforma.
"O motor que processa tudo em segundos."
2
Data Fabric — Integração + Governança
data-fabric.html · linhagem · qualidade · políticas
A malha que conecta todas as fontes com governança, linhagem e catálogo de dados. Não é onde os dados ficam — é o que garante que eles chegam certos e rastreáveis.
"A logística inteligente que conecta tudo."
1
Camada de Ingestão — Conectores / Data Lake
connectors.html · HubSpot · Shopify · Omie · APIs · CSV
Tudo que chega, bruto, sem transformação. Logs, eventos, transações, JSONs, planilhas. Preservação completa do histórico. Custo mínimo de storage (object storage).
"O depósito onde tudo começa."
D
Definições — O Que Cada Termo Significa
Glossário técnico mapeado para linguagem de negócio
Termo Definição técnica Analogia de negócio Problema que resolve
Data Lake Storage bruto de dados em qualquer formato (structured, semi, unstructured). Imutável, append-only. Reservatório de água bruta. Tudo flui pra dentro sem filtro. Nunca perder dado histórico. Custo mínimo.
Data Warehouse Dados estruturados, modelados em esquema relacional. Otimizado para queries OLAP. Arquivo organizado com gavetas etiquetadas. Fácil de buscar. Queries rápidas em dados históricos limpos.
Data Lakehouse Arquitetura híbrida: object storage + table format (Iceberg/Delta) + query engine. Melhor dos dois mundos. Armazém moderno: escala de depósito, organização de escritório. Custo de lake + performance de warehouse.
Data Fabric Arquitetura de integração distribuída com governança, catálogo e linhagem centralizados. Malha logística da empresa — conecta todo mundo com rastreamento. Silos de dados. Qualidade inconsistente. Sem rastreabilidade.
Data Mesh Paradigma organizacional: dados como produto, domínios autônomos, plataforma self-service. Descentralização — cada área é dona dos seus dados. Gargalo central. Times de dados sobrecarregados.
Semantic Layer Camada de abstração que mapeia lógica de negócio sobre SQL/modelos físicos. Single source of truth para métricas. Dicionário oficial da empresa. "Receita" significa a mesma coisa para todos. Números diferentes na mesma reunião. Planilhas conflitantes.
ETL / ELT Extract-Transform-Load (ou Load-then-Transform). Pipeline de movimentação e transformação de dados. A esteira de produção que processa a matéria-prima. Dado cru não tem valor direto. Precisa ser processado.
Data Lineage Rastreabilidade completa: de onde cada dado veio, como foi transformado, onde foi consumido. Nota fiscal de rastreabilidade de produto. Sabe exatamente a origem. Auditorias. LGPD. "Esse número veio de onde?"
A
Analogias por Setor — Fale a Língua do Cliente
A mesma arquitetura explicada com metáforas do mundo de cada cliente
PARA VAREJO / ATACADO
🏭
Data Lake = Armazém bruto
Toda mercadoria chega e é empilhada. Volume máximo, organização mínima. Barato de manter, lento de encontrar algo específico.
📦
Lakehouse = Centro de distribuição
Mercadoria organizada por SKU, separada por categoria, pronta para expedição imediata. Rápido de achar, rápido de entregar.
🔍
Semantic Layer = Código de barras
"Café 500g Torrado" significa a mesma coisa no estoque, no caixa e no relatório de compras. Uma definição, zero ambiguidade.
PARA SAAS / FINANCEIRO
🏦
Data Lake = Core bancário bruto
Cada transação registrada em log imutável. Auditoria completa. Nenhum dado deletado jamais.
📊
Lakehouse = Ledger processado
Transações agrupadas, conciliadas, disponíveis para análise em tempo real. De raw log para demonstrativo em segundos.
📋
Data Fabric = Sistema de compliance
Toda movimentação de dado rastreada. LGPD, BACEN, CVM — cada acesso, cada transformação, auditável por lei.
Analogia universal — funciona para qualquer setor
Data Lake = garimpo (tudo bruto na terra) · Lakehouse = mineradora (processa e refina) · Semantic Layer = joalheria (transforma em produto com valor) · Dashboard = vitrine (o cliente vê e compra a decisão)

Scripts por Público — O Que Dizer e Para Quem
Mesma arquitetura, três formas de apresentar — adapte pelo cargo do interlocutor
Quando usar
Identifique o cargo antes de abrir a boca. CEO/CFO = linguagem de resultado. Diretor operacional = linguagem de processo. CTO/Dev = linguagem técnica. Nunca misture os três num único pitch linear.
NÍVEL 1 — C-LEVEL · CEO, CFO, SÓCIO (30 segundos)
Script de abertura
"Hoje vocês têm dado em 5 sistemas diferentes. Nenhum conversa com o outro. A reunião de segunda começa com 20 minutos debatendo qual número é o certo. O Cortex conecta tudo e entrega um único número, sempre atual, que todo mundo confia. Do HubSpot, do Shopify, do Omie — num só lugar. Você toma a decisão. A plataforma garante que a base está correta."
Se perguntarem "como funciona?"
"Por baixo tem uma camada que puxa os dados dos seus sistemas automaticamente. Limpa, organiza e coloca num painel. Você não precisa saber os detalhes — o seu time de TI entende. O que importa pra você é que o número que aparece aqui é o mesmo que qualquer pessoa na empresa vai ver."
NÍVEL 2 — GERENTE / DIRETOR OPERACIONAL (2 minutos)
Script de processo
"O Cortex funciona em três etapas. Primeiro: conectamos suas fontes — HubSpot, Shopify, Omie, o que vocês usam. Os dados chegam aqui em tempo real, sem intervenção manual. Segundo: passam por um processo automático de limpeza e padronização — isso é o Lakehouse. Registros duplicados, formatos diferentes, campos faltando — tratados automaticamente. Terceiro: você acessa o resultado no dashboard. A diferença do que você tem hoje é que qualquer relatório que você puxar vai ter o mesmo número na mesma base."
Se perguntarem sobre o Data Fabric
"O Data Fabric é a camada de governança. Imagine que cada dado que passa pela plataforma tem um rastro — de onde veio, quando chegou, quem acessou, como foi transformado. Isso serve pra dois cenários: auditoria interna (o diretor pergunta de onde veio aquele número) e compliance regulatório (LGPD, por exemplo, exige que você saiba onde estão os dados dos seus clientes e quem acessou)."
NÍVEL 3 — TÉCNICO · CTO, ENGENHEIRO, DBA (sem limite)
Script técnico
"A ingestão acontece via conectores REST + webhook. Dado cai no Raw Zone em object storage (S3/GCS) no formato original — imutável, append-only. O pipeline ELT (dbt + Airflow) processa em batch ou near-realtime e escreve no Lakehouse com Apache Iceberg como table format. Queries via DuckDB para OLAP rápido. Em cima disso, uma semantic layer expõe as métricas curadas — mesma definição de receita pra todo mundo. O dashboard consome via API REST e GraphQL. Data lineage com OpenLineage, qualidade com Great Expectations."

Objeções Sobre Arquitetura — Como Responder
As dúvidas técnicas que viram bloqueio de venda — com respostas diretas
"Já tenho um BI. Por que preciso de Lakehouse?"
Gatilho: cliente usa Power BI, Tableau ou Looker isolado
"BI é a vitrine. Lakehouse é o estoque. O Power BI que você usa hoje está consumindo dados de onde? De planilhas? De exports manuais do ERP? O Lakehouse garante que o dado que alimenta o seu BI é automático, padronizado e histórico. Você continua com o dashboard que quer — a diferença é que o número passa a ser confiável."
"Data Fabric parece complicado demais para minha empresa"
Gatilho: empresa pequena ou média, sem time de dados
"Data Fabric é o conceito. O Cortex é a implementação prática. Você não precisa entender a arquitetura — ela já vem configurada. O que você vai notar na prática é: qualquer número que aparecer no dashboard tem origem rastreável, e qualquer dado de cliente está catalogado para fins de LGPD. É compliance automático, não uma segunda área para gerir."
"Nossos dados já estão no ERP. Para que duplicar?"
Gatilho: cliente com ERP forte (SAP, TOTVS, Omie)
"O ERP foi feito para registrar operações, não para análise. Ele é excelente no que faz. O Cortex não substitui o ERP — ele consome o ERP junto com todas as outras fontes (CRM, e-commerce, planilhas) e cria uma visão unificada que o ERP sozinho nunca vai ter. É como adicionar uma central de inteligência ao lado do seu motor operacional."
"Por que não simplesmente conectar direto no banco de dados?"
Gatilho: CTO sugere acesso direto via SQL
"Pode fazer isso. Mas três problemas vão aparecer: 1) Performance — queries analíticas pesadas no banco operacional travam o sistema. 2) Qualidade — dado bruto sem transformação gera inconsistências. 3) Histórico — bancos operacionais não guardam histórico de mudanças. O Lakehouse resolve os três sem tocar na sua infra atual."
"O que é exatamente 'camada semântica'? Parece abstrato demais"
Gatilho: cliente técnico ou financeiro questionando o termo
"Exemplo concreto: sua área de vendas calcula receita excluindo devoluções. O financeiro inclui devoluções até o boleto ser emitido. O CEO quer saber a receita líquida confirmada. São três cálculos diferentes para a mesma palavra. A camada semântica define uma vez, para todos: 'Receita = X'. Todo relatório, todo dashboard, toda query da empresa usa essa mesma definição. Acabou a briga de número em reunião."
